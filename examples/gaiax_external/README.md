# External Gaia-X Reference Credentials

This folder holds **third-party Verifiable Credentials issued by the Gaia-X
Digital Clearing House (GXDCH)**. They are **not generated by Harbour** — they
are kept as a real-world reference: a contrast against the Harbour-issued
examples in [`../gaiax/`](../gaiax/), and a record of compliance findings
observed against actual Gaia-X tooling output.

> ⚠️ **Do not use these as a copy/paste template.** See the findings below —
> notably the `digestSRI` encoding does not conform to the spec it cites.

## Findings

### 1. `digestSRI` is hex-encoded — not W3C SRI base64 (non-compliant)

Each `gx:CompliantCredential.digestSRI` value is **64 lowercase-hex characters**
(a SHA-256 digest encoded as hex):

```json
{
  "type": "gx:CompliantCredential",
  "gx:credentialType": "gx:LegalPerson",
  "digestSRI": "sha256-29784869cbb4b2970085ab3d22bd1fc732faabdd55f27f999d7f95ad7fc4b5a9"
}
```

The W3C **Subresource Integrity** spec — which **W3C VC Data Model 2.0**
(the `sriString` datatype, §B.3.1) and the **Gaia-X Compliance Document 25.10
§10** both explicitly defer to — defines the value as
`hash-algorithm "-" base64-value` using **standard base64 (RFC 4648 §4)**, not
hex. A compliant SHA-256 value is 44 characters ending in `=`. The same three
digests, re-encoded as standard base64 (the spec-correct form), would be:

| `gx:credentialType` | as published (hex) | spec-compliant (base64) |
|---------------------|--------------------|--------------------------|
| `gx:LegalPerson` | `sha256-29784869…fc4b5a9` | `sha256-KXhIacu0spcAhas9Ir0fxzL6q91V8n+ZnX+VrX/Etak=` |
| `gx:Issuer` | `sha256-d2ca1a53…3fdef6c` | `sha256-0soaU3unlqU9ck2ZNwzPFg3Op/yN+87B+8nbTeP972w=` |
| `gx:VatID` | `sha256-07532c26…1791b5a9` | `sha256-B1MsJscKDK+X1qCZHhkGQWN+GhUiARM0881n1ReRtak=` |

A strict `sriString` / SRI verifier rejects the hex form. Because it is emitted
by the **live clearing house** (not a hand-authored sample), this is a
**tooling-vs-spec divergence**, not a one-off — which is why Harbour does **not**
trust these example values and instead emits/verifies the standards-correct
base64 form (see [`docs/specs/references/subresource-integrity.md`](../../docs/specs/references/subresource-integrity.md)
and the `harbour.digest_sri` / `digest-sri.ts` modules, checked by
`make story`).

> The hash **values** themselves cannot be independently verified from this file:
> the three referenced input VCs (`gx:LegalPerson`, `gx:Issuer`, `gx:VatID`) are
> delivered out of band and are not included here. Only the **encoding** is
> assessed above.

### 2. Domain-type placement (consistent with the SHACL `sh:closed` fix)

The envelope `type` is just `["VerifiableCredential"]`, with the domain type
`gx:LabelCredential` carried on `credentialSubject`. This matches the structure
required by Gaia-X's closed (`sh:closed true`) SHACL shapes — the same concern
Harbour addresses by keeping `harbour.gx:LegalPerson` a **separate compliance
attestation type** rather than extending `gx:LegalPerson` (see
[`linkml/harbour-gx-credential.yaml`](../../linkml/harbour-gx-credential.yaml)
and [`docs/schema/credential-model.md`](../../docs/schema/credential-model.md)).
